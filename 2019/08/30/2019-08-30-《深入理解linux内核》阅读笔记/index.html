第二章 内存寻址

内存地址

逻辑地址：每一个逻辑地址都由一个段标示符和段偏移组成。
线性地址：也叫虚拟地址，在32位系统下，每个进程会虚拟出4G内存，其中0×00000000 到 0xBFFFFFFF是用户进程空间，0xC0000000 到 0xFFFFFFFF是内核进程空间。在64位系统下，每个进程并没有16EB( 2^64 )这么大虚拟内存，目前的x86-64架构CPU都遵循AMD的Canonical form, 即只有虚拟地址的最低48位才会在地址转换时被使用, 且任何虚拟地址的48位至63位必须与47位一致(sign extension). 也就是说, 总的虚拟地址空间为256TB( 2^48 )。 其中0x0000000000000000 - 0x00007fffffffffff(128TB)为用户空间, 0xffff800000000000 - 0xffffffffffffffff(128TB)为内核空间. 参考这里
物理地址：真实的内存地址，

段寄存器

intel处理器有两种方式进行地址转换：实模式、保护模式。
段寄存器的唯一目的是为了存放段选择符。分别为cd,ss,ds,es,fs和gs六个。
六个寄存器有三个有特殊的用途：
cs：代码段寄存器，指向包含程序指令的段，其中包含一个两位的字段，用以指明CPU的当前特权(CPL),0表示最高级，3表示最低级。分别称之为内核态和用户态。
ss：栈段寄存器，指向当前程序栈的段
ds：数据段寄存器，指向静态数据或全局数据段。

段描述符

每个段有一个8字节的段描述符，段描述符防在GDT中。通常只定义一个全局描述符表（GDT），单处理器只有一个GDT，多处理器每一CPU对应一个GDT。每个进程除了要存放在GDT中的段之外如果还需要创建附加的段，就可以有自己局部描述符表（LDT local descriptor table）。GDT在主存中的地址和大小存放gdtr控制寄存器中，LDT的地址和大小存放在ldtr控制寄存器。

Linux中的分段

  • 当从用户态切换到内核态时，Linux总是确保ss段寄存器装有内核数据段的段选择符。
  • 当对指向指令或数据结构的指针时，内核不需要为其设置逻辑地址的段选择符，因为cs里面就有当前的段选择符。在内核态执行的段只有一种，叫作代码段，由宏KERNEL_CS定义，当cpu切换到内核态时，只需要将宏KERNEL_CS装载进cs就可以了。

Linux中的分页

  • 前面的分段单元把逻辑地址转换为线性地址，而分页单元将线性地址转换成物理地址。页内部连续的线性地址被映射到连续的物理地址中。
  • 页单元把所有的RAM分成固定长度的页框（物理页）。每一个页框包含一个页，也就是说页框的长度与一个页的长度一致。
  • 线性地址分为目录（directory）（10bits）、页表（table）(10bits)、偏移量（offset）(12bits)三个域。线性地址内的目录字段决定页目录中的目录项，而目录项执行适当的页表，地址中的table字段又决定页表中的表项，而表项含有页所在页框的物理地址。

物理内存布局

在初始化阶段，内核必须建立一个物理地址映射来指定哪些物理地址对内核可用哪些不可以。
内核将下列页框记为保留：

  • 在不可用的物理地址范围内的页框。
  • 含有内核代码和已初始化的数据结构的页框。

保留页框中的页绝不能动态分配或交换到磁盘上。

第三章 进程

  • 子进程对一个内存单元的修改对父进程是不可见的，反之亦然。
  • 进程之间还可能存在除了父子、兄弟关系之外的其他关系：一个进程可能是一个进程组或登录回话的领头进程，也可能是一个线程组的领头进程，还可能跟踪其他进程的执行。

pidhash表及链表

顺序扫描进程链表并检查进程描述符的pid字段会非常低效，为了加快查找，引入了4个散列表。需要4个散列表是因为进程描述符包含了表示不同类型的PID的字段，而且每种类型的PID都需要自己的散列表。

等待队列的操作

在唤醒等待队列中的进程时，总是先唤醒非互斥的进程再唤醒互斥的进程，如果链表不为空的话。同时，一个等待队列同时包含互斥进程和非互斥进程的情况是非常罕见的。

进程切换

硬件上下文

虽然每个进程都有属于自己的地址空间，但是所有进程西部共享CPU寄存器。在恢复一个进程的执行之前，内核必须确保每个寄存器都装载了挂载进程时的值。进程恢复执行前装入寄存器的一组数据成为硬件上下文，一部分存放在TSS（任务状态段，task status segment）段，剩余部分存放在内核态堆栈中。
早期的Linux版本利用far jmp指令跳转到切换进来的线程的TSS描述符的选择符来执行进程切换。这条指令即修改cs寄存器，也修改eip寄存器。
Linux2.6基于几下原因使用软件进行进程切换。

  • 通过一组MOV指令逐步执行切换，可以较好控制装入数据的合法性。可以检查ds和es段寄存器的值，这些值有可能被恶意伪造。

进程切换只发生在内核态。在执行进程切换之前，用户态进程使用的所有寄存器的内容保存在内核态的堆栈上。

thread字段

每次进程进程切换时，硬件上下文被保存在进程描述符的thread_struct的thread字段中。

内核线程

内核线程有两个方面不同于普通进程：

  • 内核线程只运行在内核态，普通进程既可以运行在内核态，又可以运行在用户态。
  • 因为内核态线程只运行在内核态，它们只是用大于PAGE_OFFSET的线性地址空间。另一方面，不管在用户态还是在内核态，普通进程可以用4G的线性地址空间。

第四章 中断和异常

中断分为同步中断和异步中断：

  • 同步中断（异常）是当指令执行时由CPU控制单元产生的，之所以称为同步，是因为只有在一条指令终止执行后CPU才会发出终端。异常是由程序的错误产生的，或者是由内核必须处理的异常条件产生的。
  • 异步中断（中断）是由其他硬件设备依照CPU时钟信号随机产生的。中断是由间隔定时器和I/O设备产生的。

中断信号的作用

中断信号提供一种特殊的方式，使处理器转而去运行征程控制流之外的代码。当一个中断信号到达时，CPU必须停止正在做的事，且切换到新的活动。此时，需要在内核态堆栈中保存程序计数器的当前值（eip和cs寄存器的值），并把与中断类型相关的一个地址放入程序计数器。
在临界区中，终端必须被禁止。

中断和异常

  • 中断：
    • 可屏蔽中断：
      要根据中断允许标志的设置来判断CPU是否能响应中断请求。
    • 非屏蔽中断：不受中断允许标志位的影响，不能用软件进行屏蔽。只有几个危急事件才引起非屏蔽中断，且总是由CPU进行辨认。
  • 异常：
    • 处理器探测异常：通过CPU控制单元产生异常时保存在内核态堆栈eip的值可分为三组。
      • 故障：保存在eip中的值是引起故障的指令地址，在异常处理程序终止时，那条指令会被重新执行。
      • 陷阱：保存在eip中的值是一个随后要执行的指令地址，只有当没有必要重新执行已终止的指令时，才触发陷阱。陷阱的主要用途是为了调试程序。
      • 异常中止（abort）
    • 编程异常：
      在编程者发出请求时发生。有int或int或int3指令触发。编程异常通常也被称为软中断。

中断描述符表

  • 中断描述符表（interrupt descriptor table IDT）是一个系统表，它与每个中断或异常向量相联系，每一个向量在表中有相应的中断或异常处理程序的入口地址。
  • 表中的每一项对应一个中断或异常向量，每个向量8字节，最多需要256x8 = 2048字节来存放IDT。
  • idtr CPU寄存器存放IDT的线性基地址及其限制。
  • IDT包含三种类型的描述符：任务门、中断门、陷阱门。
  • Linux利用中断门处理中断，陷阱门处理异常。

    中断和异常处理程序的嵌套执行

  • 与异常相关的至多两个内核控制路径会堆叠在一起（第一个系统调用引起，第二个由缺页引起）。因为“Page Fault”异常处理程序从不进一步引起异常。
  • 一个终端处理程序即可以抢占其他的中断处理程序，也可以抢占异常处理程序。相反，异常处理程序从不抢占中断处理程序。

多种类型的内核栈

内核栈位于thread_union中，根据编译内核的选项不同，tnread_union结构可能占据一个页框或两个页框。若tnread_union占据两个页框(8KB)，那么当前进程的内核栈就被用于所有类型的内核控制路径。若tnread_union结构占据一个页框（4KB），内核只使用三种内核栈：

* 异常栈：用于处理异常，包括系统调用。
* 硬中断请求栈：用于处理中断。
* 软中断请求栈：用于处理可延迟函数。

为中断处理程序保存寄存器的值

内核用负数来表示所有的中断，因为正数用来表示系统调用。

do_IRQ()函数

关键字regparm表示函数到eax寄存器中去找到参数regs的值。

挽救丢失的中断

在多处理器系统中，假设有一条激活的IRQ线，在CPU应答中断前，这条IRQ线被另一个CPU屏蔽掉；结果IRQ_DISABLED标志被设置。随后，当前的CPU开始处理挂起的中断，但是do_IRQ()函数检测到IRQ_DISABLED标签被设置，所以不会执行中断服务例程，造成中断丢失。在这种情况下，内核用来激活IRQ线的enable_irq()函数先检测是否发生了中断丢失，如果是，该函数就强迫硬件让丢失的中断再产生一次。

第七章 进程调度

Linux是抢占式多任务系统，进程的抢占只发生在用户态执行阶段，在内核态执行时是不能抢占的。
Linux采用了两种不同的优先级范围：

  • nice值：nice值是所有UNIX系统中的标准化概念，但是不同的UNIX系统由于调度算法不同，nice值的运用方式也有所差异。nice值的范围从-20～+19，默认为0；越大的nice值意味着更低的优先级，相比高nice值的进程，低nice值的进程可以获得更多的处理器时间。
  • 实时优先级：实时优先级的值是可以配置的，默认情况下它的变化范围是0～99（包括0，99）。与nice值相反，实时优先级值越高意味着进程优先级越高，任何实时进程的优先级都高于普通进程。实时优先级和nice优先级处于两个互不相交的范畴。

第八章 内存管理

RAM的某些部分永久地分配给内核，用来保存内核代码以及静态内核数据结构。其余部分被称为动态内存。整个系统的性能取决于如何有效的管理动态内存。因此，现在所有多任务操作系统都在尽力优化对动态内存的使用，即用时分配，不用时释放。

页框管理

Linux采用4KB页框大小作为标准的内存分配单元。

页描述符

内核必须能够区分哪些页框包含的是属于进程的页，哪些页框包含的是内核代码或内核数据。页框的状态存放在page的页描述符中，所有的页描述符存放在mem_map数组中，每个描述符的长度为32字节，所以mem_map所需要的空间略小于RAM的%1。

  • _count:
    页的引用计数器。如果该数为-1，则代码该也空闲，可被分配给任一进程或内核本身。page_count()函数返回_count加1后的值，也就是该页的使用者的数量。

    非一致内存访问（Non-Uniform Memory Access,NUMA）

    在非一致内存访问模型，在这种模型中，给定CPU对不同内存单元的访问时间可能不一样。系统的物理内存被划分为几个不同节点。在一个单独的节点中，任一给定的CPU访问页面所需的时间都是相同的。

    内存管理区

    Linux内核必须处理80x86体系结构两种硬件约束：
  • ISA总线的直接内存存取（direct memory access,DMA）处理器有一个严格的限制：只能对RAM前16MB寻址。
  • 在具有大容量RAM的现代32/64位计算机中，CPU不能直接访问所有的物理内存，因为线性地址空间太小。

为了应对这两种限制，Linux2.6把那个内存节点的物理内存划分为3个管理区（zone）。
在80x86 UMA（uniform memory access）体系结构的管理区为：

  • ZONE_DMA: 包含低于16MB的内存页框。
  • ZONE_NORMAL: 包含高于16MB且低于896MB的内容页框。
  • ZONE_HIGHMEM: 包含从896MB开始高于896MB的内存页框。

ZONE_DMA和ZONE_NORMAL区包含内存的“常规”页框，通过把它们线性地址映射到线性地址空间的第4个GB，内核就可以直接进行访问。ZONE_HIGHMEM区包含的内存页不能由内核直接访问，尽管它们也线性地映射到了线性地址空间的第4个GB。在64位体系结构上ZONE_HIGHMEM区总是空的。

当内核调用一个内存分配函数时，必须指明请求页框所在的管理区。

保留的页框池

有两种不同的方法来满足内存分配请求。如果内存有足够的空间可用的话，请求就会被立刻满足。否则，必须回收一些内存，并且将发出请求的内核空中路径阻塞，直到有内存被释放。
在某些情况下，内核控制路径不能被阻塞，比如，处理中断或执行临界区代码时。在这些情况下，一条内核控制路径应当产生原子内存分配请求。原子请求从不阻塞；如果没有足够的空闲页，则仅仅是分配失败。
内核为了尽量减少原子内存分配失败事件的发生，为原子内存分配请求保留了一个页框池，只在内存不足的时候使用。 保留内存的数量（以KB为单位）存放在min_free_kbytes变量中。在内核初始化时设置，并取决于ZONE_DMA，ZONE_NORMAL内存管理区的页框数量。

高端内存页框的内核映射

896MB边界以上的页框并不映射到内核线性地址空间的第4个GB，因此，内核不能直接访问它们。
在64位硬件平台上不存在这个问题，因为可使用的线性地址空间远大于能安装的RAM大小。这些体系结构的ZONE_HIGHMEM总是空的。但是在32位下，不得不采用以下方法来允许内核使用所有可使用的RAM，来达到PAE支持的64GB。

  • 高端内存页框的分配只能通过alloc_pages()函数和它的快捷函数alloc_page()。这些函数不返回第一个被分配页框的线性地址，因为如果该页框属于高端内存，那么这样的线性地址根本不存在。这些函数返回第一个被分配页框的页描述符的线性地址。这些线性地址总是存在的，因为所有的页描述符一旦分配在低端内存中，它们在内核初始化阶段不会改变。
  • 没有线性地址的高端内存中的页框不能被内核访问。因此，内核线性地址空间的最后128MB中的一部分专门用于映射高端内存页框。这种映射只是暂时的，否则只有128MB的高端内存被使用。取而代之，通过重复使用线性地址，使得整个高端内存能够在不同的时间被访问。

内核采用三种不同的机制将页框映射到高端内存：分别叫做永久内核映射、临时内核映射和非连续内存分配。
建立永久内核映射可能阻塞当前进程；永久内核映射不能用于中断处理程序和可延迟函数。相反，建立临时内核映射决不会要求阻塞当前进程。

伙伴算法

伙伴算法的主要目的是减少外部碎片，同时运行快速分配和回收物理页面。伙伴算法分配依赖的数据结构为free_area数组，该数组有11个元素，每个数组元素为相同空闲块大小的链表的表头，对应的链表链接的空闲块的大小为2^order个页面大小，order为数组下标。free_area数组示例图如下：

比如当需要分配4个页面大小的内存卡时，首先去free_area[3]找，若该链表上有空闲块（4个页面）则分配给用户，否则向free_area[4]去查找，如果有空闲块（8个页面）则分配，并将该空闲块分裂成两个4个页面大小的空闲块，将前4个页面的块链接到free_area[3]，后4个页面大小的块返回给用户。否则继续往下，知道查询到最后一个都不满足，返回分配失败。

0x00 前言

这篇文章是看了大佬的文章，总结的一些笔记。

0x01 BPF(Berkeley Packet Filter)模块简介

BPF(Berkeley Packet Filter)模块是用于支持用户态自定义包过滤方法的内核模块，目前各大Linux发行版都默认开启了bpf支持。

本质上它是一种内核代码注入的技术：

  1. 内核中实现了一个cBPF/eBPF虚拟机；
  2. 用户态可以用C来写运行的代码，再通过一个Clang&LLVM的编译器将C代码编译成BPF目标码；
  3. 用户态通过系统调用bpf()将BPF目标码注入到内核当中；
  4. 内核通过JIT(Just-In-Time)将BPF目编码转换成本地指令码；如果当前架构不支持JIT转换内核则会使用一个解析器来模拟运行，这种运行效率较低；
  5. 内核在packet filter和tracing等应用中提供了一系列的钩子来运行BPF代码。
    它的优点在于提供了一种在不修改内核代码的情况下，可以灵活修改内核处理策略的方法。

0x02 漏洞简介

该漏洞存在于BPF模块。整数溢出漏洞存在于BPF_MAP_CREATE功能中，是属于bpf()系统调用的一部分。该漏洞影响linux kernel 4.20rc1~4.20rc4，这篇文篇是基于linux kernel 4.20 rc3来进行复现的。

0x03 漏洞分析

该漏洞的根本成因是在创建queue_stack_map时存在整数溢出导致申请内存偏小，在map_update_elem系统调用中可以向这块map拷贝数据导致内核堆溢出。

a. 整数溢出

整数溢出漏洞存在queue_stack_map_alloc()函数中，具体的函数调用栈如下：

从bpf()系统调用一步一步往下分析。

  • bpf()系统调用

bpf（）系统调用执行的操作由cmd参数确定。每个操作都有一个附带的attr参数,它是一个指向bpf_attr类型的联合的指针。 size参数是attr指向的union的大小。

  • map_create（）

map_create()函数会创建一个map结构体，并分配一个文件描述符，之后可以用这个文件描述符来访问。map是一个全局变量。

  • find_and_alloc_map

find_and_alloc_map()首先会根据传入参数获取type值。eBPF支持多种不同类型的map，所以用户在创建map的时候，需要根据应用场景指定一个合适的类型。目前版本支持的map类型定义在文件include/uapi/linux/bpf.h，如下：

这里漏洞存在于queue_stack_map_alloc（），所以这里的type值为23。然后根据的type值找到对应的虚函数表。
在bpd_types.h文件中看到BPF_MAP_TYPE_QUEUE被定义成queue_map_ops。

进一步跟进queue_map_ops。可以看到：

map_alloc()对应虚函数queue_stack_map_alloc()，也就是我们的漏洞函数。
其中结构体qs相当于分配的内存块的头部，之后才是数据部分。

  • queue_stack_map_alloc

key_size和value_size分别用于指定map存储的键值大小，而max_entries则指定这个map对象中可以存储的键值对数量。这里的size的类型是u32，若max_entries=0xffffffff，size = attr->max_entries + 1之后size的值为0，queue_size = sizeof(*qs) + (u64) value_size * size;就相当于queue_size = sizeof(*qs);。

sizeof(*qs)相当于用于管理这块内存的头部的大小，而value_size*size为size个entry加起来的总的内存大小，每个entry的大小为value_size，也就是之后数据存放的地方。

这样相当于只申请了头部大小的堆块，导致申请的堆块偏小，若之后有对这块内存进行拷贝等操作，就会导致堆溢出，且bpf_map结构体的key_size和value_size都是用户可控。

b. 堆溢出

由于上面创建的map属于全局变量，其他的程序，进程可以根据之前创建时生成的文件描述符定位到相应的map对象进行一些操作，比如查询、更新和删除等。由于在创建这块map对象时申请的内存块比实际要申请的内存块要小，导致在进行push操作时会导致溢出。堆溢出漏洞存在BPF_MAP_UPDATE_ELEM系统调用中。
函数调用栈如下：

  • map_update_elem
    map_update_elem（）会根据参数中获取文件描述符找到对应的map对象，并且根据map对象中的value_size申请一块内存，并且将用户输入拷贝到这块新申请的内存当中。然后根据map对象中的type值找到对应的虚函数表，并调用对应的虚函数进行处理。
  • queue_stack_map_push_elem
    之前创建的map对象的类型是BPF_MAP_TYPE_QUEUE，所以这里会调用map_push_elem(),对应的函数为queue_stack_map_push_elem。
    bpf_queue_stack结构体：

这个结构体就是之前创建map对象时分配的区域。每一个map对象中有个size个entry，每个entry的大小为value_size，每次调用queue_stack_map_push_elem()可以往一个entry里面写入内容。

堆溢出位于memcpy(dst, value, qs->map.value_size);处，dst堆块在创建时存在整数溢出，导致只有sizeof(struct bpf_queue_stack)大小。

0x04 漏洞利用

堆风水

需要在溢出以前完成堆风水，就是将特定的内存块分配到特定的地址。根据Linux内核的堆分配机制，在分配堆块时，相同大小的堆块申请一块大内存，相同大小的堆块相邻，思路是要找一个包含函数指针或写指针的指针放到溢出堆块后面，上面分析看到，struct bpf_queue_stack的第一个成员bpf_map_ops就是一个包含许多函数指针的虚表指针，我们只需要连续申请两个bpf_queue_stack，就可以让第一个bpf_queue_stack发生溢出，改写后一个bpf_queue_stack的虚表指针。

可以通过溢出将ops指向一块伪造的虚表。close()bpf map时，会将bpf_map_free_deferred()添加到队列并随后执行，通过将map->ops指向用户态可控位置，并且将ops.map_free设为任意值，我们就可以在执行map->ops->map_free(map);语句时将rip设置为任意值。

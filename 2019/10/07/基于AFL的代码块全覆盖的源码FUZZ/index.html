0x00 总体思路

在AFL的基础上进行改进，AFL是基于边覆盖率的，没有考虑基本块覆盖。改进的想法是做基本块覆盖，使之在fuzzing的过程中可以做到基本块全覆盖。如何实现基本块全覆盖，初步想法是将fuzz分为两个阶段，以下是具体实现思路：

  1. 第一阶段，沿用afl的fuzzing流程。将所有能执行到新的基本块样例重新放回输入队列进行变异，并实时监测代码覆盖率，若在一个有限的时间内，代码覆盖率趋于稳定不再变化，则进入第二阶段。用一块共享内存bitmap来记录基本块的命中与否，以bit为单位，因为不记录基本块的命中次数，只记录是否命中。源码插桩统计的基本块数n，共享内存的大小为n/8B。
  2. 第二阶段，该阶段针对第一阶段没有执行到的基本块进行fuzzing。可以通过遍历bitmap得知基本块的命中情况，将在第一轮fuzzing过程中没有命中的基本块拿出来，分析出执行到这些基本块的所有的控制流依赖条件语句（就是能到达基本块的路径上的所有条件语句），并通过动态污点分析，确定与这些条件语句相关的输入字节，并且在变异的过程中保证目标基本块可达。（现在很多采用污点分析fuzz工具都有一个通病就是效率很低，通常比没有加污点分析的fuzz工具要慢十几倍。之所以会这么慢，是因为一个稍微复杂一点的程序，基本块的数量也是非常多的，这样全部分析就会导致fuzz的速度很慢。因为要做到基本块全覆盖，所以我只对在第一轮没有执行到的基本块进行污点分析，这个分析的基本块的数量会大大减少。）这里循环fuzzing剩下基本块的时候，在分析出所有基本块的先验条件语句，从路径最短的开始变异。

0x01 模块

具体实现可以分为7个模块

  1. 输入变异模块。之后会考虑污点分析，afl的变异策略有确定性策略和随机性策略两种。在不考虑污点分析的情况下，沿用afl的变异策略，但是这样的效率比较低下，因为数量会非常大。如果加入了污点分析，在进行基本块的定位时，只去变异与条件语句有关的比特，这样会大大加快变异效率。在已经执行到目标基本块后，可以沿用afl的变异策略。
  2. 语料库迭代部分：对变异种子的选择，在afl中，是将能执行到新的tuple的样例重新放回输入队列，继续变异。但是这种效率不是很高，这里需要想一个新的种子迭代算法，来提升效率和基本块覆盖率。将afl存储分支信息是用一个64kb的共享内存，之所以是用byte而不是bit，是因为还要记录tuple命中数。改进思路是在插桩时，统计出基本块的数量n，然后申请一个共享内存大小为nbit，用一个bit来记录基本块的命中与否，不记录基本块的命中次数。然后只将能执行到新的基本块的测试样例重新放到语料库继续变异。
  3. 源码插桩模块： 利用源码插桩来获取基本块覆盖率，和得到基本块的支配关系。还要能得到目标基本块的控制流条件语句（控制流依赖关系：post-dominator tree 来实现（llvm 提供了程序内的post-dominator tree 的库，没有提供程序间的post-dominator tree ）。要得到目标基本块的控制流依赖条件语句。

  4. 静态分析/污点分析模块：分析输入中的哪些字节是流向目标基本块的控制条件语句。得到目标基本块的目标条件语句的数据流依赖条件语句。

    • 比如下面这个样例，输入的前4字节控制x,输入后4字节控制y，那么3行中是目标基本块。该目标基本块对应的目标条件语句是行2，哪些污点分析就是要分析出输入中的哪几个字节影响x,y。然后在fuzzing的第二个阶段指对这些字节进行变异。

  1. fuzz模块：核心模块，初步计划是沿用afl的fuzz算法，这个地方需要加以创新。这里初步想法是，整个fuzzing过程分为两个阶段，前提条件是，有一块共享内存，用于来记录所有基本块的执行情况。
    PS: : afl对种子进行fuzz时，在使用随机策略时，随机选择一系列的变异策略并应用到种子输入的随机位置；在确定性变异策略阶段，afl在输入的每个字节上进行某些变异策略。
  2. 覆盖率统计模块：统计基本块覆盖率，并将其结果反馈给fuzz模块，选择下一步的fuzz策略。
  3. monitor：检查程序的状态，记录crash，并过滤掉重复的crash。

具体计划

具体计划分为以下几个大步骤：

  1. 第一阶段的实现：
    a. 前面一步需要得到程序的基本块的总数，要通过源码插桩实现。分析afl-as部分，如果afl已经实现了在所有的基本块插入探针，则这步就省略。得到基本块的总数为BLOCK_NUMBER，每一个基本块有一个范围在0~BLOCK_NUMBER的随机数来唯一标识。（这里暂时不考虑基本块数量爆炸的问题，因为有些复杂的程序，基本块数量会非常多，导致共享内存需要很大，申请内存会失败）
    b. 修改afl代码共享内存部分：映射一块共享内存bitmap，来记录基本块的命中与否，在整个fuzz的过程中如果有测试样例执行到了该基本块，将对应的为置为1，否则为0。
    c. 修改afl代码中tuple信息记录部分。将原来代码中的通过前一个基本块的key和当前基本块的key做异或的边的部分改成直接记录基本块的命中处理。
    d. 设置一个stable_time。若代码覆盖率在这个时间段内保存不变，或者只有很小的波动。则进入第二个阶段。
  2. 第二阶段的实现：
    a. 定位在第一阶段没有执行到的基本块。
    b. 循环遍历上面的基本块，获取其控制流依赖条件语句。（这里面涉及很多细节，设计很多技术，还不清楚，要进一步去调研）
    c. 循环遍历上面的基本块，获取其数据流依赖条件语句。（很多细节，很多要用到的技术，待进一步调研）
    d. 找到一种可以获取全局最小值的算法，解约束。（机器学习里面有很多）
    e. 制定变异策略。根据前面的控制流依赖和数据流依赖，指定恰当的变异策略满足约束，使得在第一阶段没有执行到的基本块全部可达。

污点分析学习笔记

欠污染：由于对隐式流污点传播处理不当导致本应被标记为污点的变量没有被标记的问题。
过污染：污点标记过多而导致污点变量大量扩散的问题。
无害处理：是指污点数据在经过无害处理后，污点数据本身不再携带敏感信息或者针对该数据的操作不会再对系统产生危害。换言之，污点数据在经过无害处理模块后，污点标记可以被移除。正确使用无害处理可以降低系统中的污点标记的数量，提高污点分析的效率，并且避免由于污点污点扩散导致的分析结果不精确的问题。

静态污点分析技术：是指在不运行程序且不修改代码的前提下，通过分析程序变量间的数据依赖关系来检测数据能否从污点源传播到污点汇聚点。

  1. 要根据程序中函数之间的调用关系构建调用图（控制流图）
  2. 常见的显式流污点传播方式包括直接复制传播、通过函数调用传播以及通过别名（指针）传播。
    动态污点分析技术：优点是误报率低，可信度高。缺点是漏报率较高，平台相关性高，资源消耗高。
